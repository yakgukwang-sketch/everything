# 프로젝트 개요

## 한 줄 요약

전국 모든 로컬 서비스 데이터를 통합하여 AI 에이전트들에게 API로 제공하는 데이터 인프라 플랫폼.

---

## 문제 인식

현재 온라인 플랫폼들은 각 영역의 데이터를 독점하고 높은 수수료를 부과하고 있다.

- 배달의민족, 쿠팡이츠: 30~40% (중개수수료 + 광고비 + 배달비)
- 에어비앤비: 15~20% (호스트 + 게스트)
- 네이버 스마트스토어: 10~15% (판매수수료 + 광고비)
- 야놀자, 여기어때: 10~20%

오프라인 시대 부동산 중개수수료가 0.5%였던 것에 비해, 온라인 중개는 오히려 수십 배 비싸졌다. 이는 효율의 문제가 아니라 독점의 문제다. 이 수수료는 결국 소비자 가격에 전가되어 전체 물가를 끌어올리고 있다.

또한 소비자는 영역별로 분리된 앱을 각각 설치해야 한다. 밥을 시키려면 배민, 숙소를 잡으려면 야놀자, 물건을 사려면 쿠팡, 택시를 부르려면 카카오택시. 같은 본질(로컬 자원의 연결)인데 앱이 따로 존재한다.

---

## 핵심 아이디어

AI 에이전트 시대에는 앱이라는 경계가 사라진다. 소비자는 에이전트에게 "다음 주 부산 여행 잡아줘"라고 말하면 숙소, 맛집, 교통이 한 번에 처리된다. 이때 모든 에이전트가 공통으로 필요한 것이 있다 — 로컬 데이터다.

우리는 이 데이터를 통합하여 API로 제공한다. 에이전트가 직접 데이터를 수집할 필요 없이, 하나의 API에서 모든 로컬 서비스 정보를 가져갈 수 있도록 한다. 우리는 에이전트를 만들지 않는다. 에이전트들이 사업할 수 있는 원자재를 공급한다.

---

## 공유 플랫폼의 통합

현재 로컬 서비스 플랫폼들의 공통점은 자기 자산이 없다는 것이다. 배민은 음식점을 소유하지 않고, 야놀자는 호텔을 소유하지 않고, 우버는 차를 소유하지 않는다. 전부 남의 자산을 연결해주고 수수료를 받는 공유 플랫폼이다.

이 공유 플랫폼들을 하나의 데이터 레이어로 통합한다.

**통합 대상 영역:**

- 음식/배달: 배달의민족, 쿠팡이츠, 요기요, 캐치테이블, 네이버 플레이스
- 숙박/여행: 야놀자, 여기어때, 에어비앤비, 부킹닷컴, 마이리얼트립
- 부동산: 직방, 다방, 네이버부동산
- 쇼핑: 네이버쇼핑, 쿠팡, 무신사, 오늘의집, 당근마켓
- 이동: 카카오택시, 우버, 티맵, 코레일, 스카이스캐너
- 생활서비스: 숨고, 크몽, 미소
- 병원/건강: 굿닥, 똑닥, 강남언니
- 엔터테인먼트: CGV, 인터파크, 예스24

형태만 다르지 데이터 구조는 동일하다 — 위치, 가격, 시간, 리뷰. 이 네 가지로 모든 로컬 서비스를 표현할 수 있다.

## 실제 사용 시나리오 (배달 예시)

### 1. 소비자 요청

검색창에 입력한다:
> "12시까지 4명 4만원이하로 맞춰서 점심 시켜놔. 남자 2명 여자 2명"

### 2. 사전 질문 (우리 웹페이지에서 처리)

우리 웹페이지가 소비자에게 한 번만 질문한다. 버튼을 눌러 선택하면 끝이다. 에이전트마다 각각 질문하면 소비자가 피곤해지기 때문에, 사전 질문 정제는 우리가 담당한다. 이는 소비자와 에이전트 사이의 통신 규격을 표준화하는 인프라 역할이다.

- 못 먹는 음식이 있나요? → [없음] [해산물] [돼지고기] [기타]
- 알러지가 있나요? → [없음] [땅콩] [유제품] [기타]
- 같이 먹어도 되나요? → [같이 먹기] [각자 따로]
- 덜어 먹나요? → [덜어 먹기] [1인분씩]
- 선호 장르? → [한식] [중식] [양식] [일식] [상관없음]

### 3. JSON 신호 전송

소비자의 선택을 표준화된 JSON으로 변환하여 모든 에이전트에게 동시에 전송한다.

```json
{
  "deadline": "12:00",
  "headcount": 4,
  "budget": 40000,
  "group": { "male": 2, "female": 2 },
  "restrictions": ["none"],
  "allergies": ["none"],
  "sharing": "together",
  "serving": "shared_plates",
  "preference": "any",
  "location": "강남역 2번출구 근처"
}
```

### 4. 에이전트들의 영업 (세일즈맨 경쟁)

에이전트들이 API에서 데이터를 가져와 각자의 알고리즘으로 최적 제안을 만들어 응답한다.

**에이전트 A (최저가형)**
- 39,500원 | 수수료 2%
- "동네 한식당 제육볶음 세트, 4인분에 딱 맞음"
- 배달비 2,000원 포함

**에이전트 B (취향분석형)**
- 38,000원 | 수수료 4%
- "양식 추천. 이유: 당신이 지난주에 파스타 주문 후 별점 5점 줬음. 이 가게 까르보나라 리뷰 4.7"
- 보험 포함

**에이전트 C (숨은맛집형)**
- 37,000원 | 수수료 3%
- "이 동네 숨은 맛집. 리뷰 127개 평균 4.8, 네이버 플레이스에서 '또 가고 싶다' 키워드 82회"
- 식당 협찬 할인 적용

**에이전트 D (건강특화형)**
- 41,000원 (예산 약간 초과) | 수수료 5%
- "샐러드+그릴 세트, 1인당 580kcal. 단백질 35g"
- 칼로리 상세 제공

### 5. 소비자 선택

소비자가 에이전트 B를 선택하고 터치한다. 에이전트 B는 영업 성공.

### 6. 이후 과정 (전부 에이전트 자율)

음식점 주문, 라이더 매칭, 결제, 보험 — 모든 과정을 에이전트가 알아서 처리한다.

라이더 매칭의 경우, 음식점 근처의 라이더들에게 여러 배달 에이전트가 경쟁적으로 제안을 보낸다.

```
배달에이전트 X: 1.2km | 3,500원 | 한집배달 O
배달에이전트 Y: 1.2km | 3,000원 | 한집배달 X (묶음배달)
배달에이전트 Z: 1.2km | 3,200원 | 한집배달 O | 보험포함
```

라이더가 자신에게 가장 유리한 조건을 선택한다.

우리가 하는 일은 하나다. 에이전트가 API를 호출하면 데이터를 제공하는 것. 그 데이터를 어떻게 조합하고, 어떻게 소비자에게 보여주고, 어떻게 주문을 처리할지는 전적으로 에이전트의 몫이다.

### 7. 평가

배달 완료 후 소비자가 에이전트를 평가한다. 이 평가가 쌓여서 다음 소비자의 선택 기준이 된다.

---

### 데이터 레이어 (우리)

전국 로컬 서비스 데이터를 크롤링·수집하여 통합 데이터베이스를 구축한다. 네 가지 레이어로 구성된다.

1. **위치 레이어**: 장소, 좌표, 주소
2. **카탈로그 레이어**: 상품, 서비스, 메뉴, 가격
3. **평판 레이어**: 전 플랫폼 리뷰 통합 (AI 요약 기반)
4. **가용성 레이어**: 실시간 예약 가능 여부, 재고, 영업 상태

이 데이터를 API로 제공한다.

### AI 에이전트 (외부)

수많은 AI 에이전트가 우리의 API를 호출하여 소비자에게 서비스를 제공한다. 각 에이전트는 자체 알고리즘과 전략으로 경쟁한다.

- 에이전트 A: 최저가 추천, 수수료 2%
- 에이전트 B: 맛집 큐레이션, 수수료 4%, 리뷰 분석 특화
- 에이전트 C: 건강식 전문, 칼로리 계산 포함
- 에이전트 D: 식당 협찬 기반, 할인가 제공

에이전트들은 각각의 세일즈맨이다. 소비자는 자기 취향과 예산에 맞는 에이전트를 선택한다. 결제 방식(카드, 크립토), 보험 포함 여부, 배달 속도 — 이 모든 것을 에이전트가 자율적으로 결정하고, 그에 따라 소비자가 지불하는 금액이 달라진다.

### 라이더/서비스 제공자 (외부)

배달의 경우, 별도의 AI 에이전트가 라이더 풀에 접속하여 경쟁적으로 배달 건을 제안한다. 라이더는 자신에게 가장 유리한 조건을 직접 선택한다. 라이더에게는 신용 시스템을 통해 결제 수단을 부여하며, 배달 이력이 쌓일수록 한도가 늘어나는 구조다.

### 소비자 인터페이스

웹페이지는 구글처럼 검색창 하나. 소비자가 검색하면 여러 에이전트의 결과가 비교되어 보여진다. 에이전트들은 각자 홍보 문구를 올리고, 소비자가 사용 후 평가한다. 우리는 이 정보를 조작하지 않고 투명하게 보여준다. 우리의 역할은 공정한 경기장을 제공하는 심판이다.

## 케어 시스템

AI 에이전트 시대에는 중앙 통제가 없기 때문에 문제 발생 시 소비자 보호가 중요하다. 그러나 케어(보험)는 우리가 직접 제공하지 않는다. 에이전트의 자율 영역이다.

각 에이전트가 보험 포함 여부, 보험료 수준, 보험사 제휴를 자체적으로 결정하고 경쟁한다.

- 에이전트 A: "보험 포함, 수수료 5%, 전액 환불 보장"
- 에이전트 B: "보험 없음, 수수료 2%, 최저가"
- 에이전트 C: "프리미엄 보험, 수수료 6%, 30분 지연 시 자동 보상"

소비자가 자신의 필요에 맞는 에이전트를 선택한다. 보험까지 우리가 제공하면 사업이 무거워지고 기존 플랫폼과 다를 게 없어진다. 우리는 데이터 API만 제공한다는 원칙을 지킨다.

---

## 수익 모델

1. **데이터 API 사용료**: 에이전트의 API 호출당 과금. 에이전트 수가 늘어날수록 자동 성장.
2. **웹페이지 트래픽**: 에이전트 등록 및 프리미엄 노출, 인증 배지 등.

두 가지 모두 거래량에 비례하며, 고정비는 데이터 수집과 서버 유지 정도로 극도로 낮다. 에이전트가 누가 이기든 상관없다. 모든 에이전트가 우리의 API를 호출해야 하므로 생태계가 커질수록 수익이 자동으로 따라온다.

## 경쟁 우위

**데이터 선점이 곧 해자다.** 전국 로컬 서비스 데이터를 먼저 통합한 사업자가 표준이 된다. 후발 주자가 같은 규모의 데이터를 재구축하는 것은 극도로 어렵다.

**극도로 가벼운 구조.** 기존 플랫폼은 영역 하나 확장할 때마다 영업팀, 운영팀, CS팀, 법무팀이 필요하다. 우리는 데이터 API만 제공하므로 엔지니어 중심 소규모 팀으로 운영 가능하다. CS, 라이더 관리, 음식점 계약 — 이 모든 운영 부담이 없다.

**규제 리스크가 낮다.** 배달앱 수수료 상한제 입법이 논의되고 있지만, 우리 모델은 오히려 수수료를 낮추는 구조이므로 정부 기조와 방향이 같다. 규제가 아닌 경쟁으로 문제를 해결한다.

**영역 확장이 무한하다.** 배달에서 시작하지만, 같은 API 구조를 숙박, 쇼핑, 이동, 생활서비스 전체로 복제할 수 있다. 데이터의 본질(위치, 가격, 시간, 리뷰)이 동일하기 때문이다.

**네트워크 효과.** 에이전트가 많아지면 소비자가 모이고, 소비자가 모이면 에이전트가 더 들어오고, 거래가 늘면 데이터가 쌓여서 케어와 추천이 정밀해진다. 이 선순환이 시작되면 가속이 붙는다.

---

## 사회적 가치

독점 플랫폼이 가져가던 수수료(30~40%)가 경쟁 구조(3~5%)로 내려가면, 그 차이가 소비자와 소상공인에게 돌아간다. 진입장벽이 사라지면서 공급이 늘어나고, 공급이 늘어나면서 소비자 가격이 내려가는 선순환이 발생한다.

플랫폼 수수료의 해체는 곧 물가 안정이다. 전국의 모든 로컬 거래에서 독점 마진이 빠지면 경제 전체에 긍정적 영향을 미친다.

## 최종 목표: 자율주행 로봇 배달

배달 영역의 최종 목표는 라이더 없는 배달이다.

현재 배달 비용의 가장 큰 부분은 라이더 인건비다. AI 에이전트가 라이더를 매칭하는 것은 과도기적 단계일 뿐이다. 최종적으로는 자율주행 로봇이 배달을 수행한다.

이 단계에 도달하면 배달 비용이 전기료와 로봇 유지비 수준으로 떨어진다. 라이더 인건비, 라이더 보험, 라이더 관리 — 이 모든 비용이 사라진다. 소비자가 체감하는 배달비가 현재 3,000~4,000원에서 수백 원 수준까지 내려갈 수 있다.

우리의 데이터 인프라는 이 전환에 필수적이다. 자율주행 로봇이 배달하려면 음식점 위치, 실시간 주문 상태, 최적 경로, 소비자 위치 — 이 모든 데이터가 실시간으로 연동되어야 한다. 우리의 API가 그 데이터를 제공한다.

단계적 전환 경로:

1단계 — 라이더 매칭: AI 에이전트가 프리랜서 라이더 풀에서 최적 매칭
2단계 — 라이더 + 로봇 혼합: 단거리는 로봇, 장거리는 라이더
3단계 — 완전 자율주행: 로봇이 모든 배달 수행

이 전환이 완료되면 배달이라는 행위의 한계비용이 거의 0에 수렴한다. 배달 수수료의 소멸을 넘어, 배달 비용 자체가 소멸하는 것이 최종 비전이다.

## 비전

AI 에이전트 시대의 로컬 데이터 인프라. 데이터는 우리가 깔고, 운영은 AI가 하고, 경쟁은 에이전트가 한다. 독점 플랫폼이 쥐고 있던 권력을 경쟁 구조로 분산시키면서, 우리는 그 경쟁의 인프라를 제공한다.
