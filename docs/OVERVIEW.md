# everything 프로젝트 개요

## 한 줄 요약

전국 모든 로컬 서비스 데이터를 통합하여 AI 에이전트들에게 API로 제공하는 데이터 인프라 플랫폼.

## 핵심 비유: 안드로이드

기존 플랫폼(배민, 쿠팡이츠, 야놀자)은 **아이폰**이다. 데이터도, 알고리즘도, 앱도 전부 자기가 만들고 운영한다. 1개의 세일즈맨이 모든 소비자를 상대한다.

우리는 **안드로이드**다. 운영체제(데이터 인프라 + 결제 + 웹)만 만들고, 그 위에서 수많은 에이전트(제조사)가 각자의 방식으로 서비스한다. 안드로이드가 스마트폰 시장을 열었듯, 우리는 AI 에이전트 시장을 연다.

---

## 구조

```
소비자 → 웹/앱 → 에이전트 경쟁 → 데이터 API → 크롤링 + 사업자 직접 등록
         (우리)    (외부)           (우리)         (양방향 수집)
```

### 배달 서비스 흐름
```
소비자 → 웹/앱 → 에이전트 경쟁 → 기사 매칭 → 배달 완료 → 평가
                                    ↑
                    가게 데이터 (네이버 플레이스 + 카카오맵 교차검증)
```

---

## 배달 서비스 흐름 (Food Delivery Flow)

```
소비자: "부천 제육볶음 4인분 4만원 안에 시켜줘"
    ↓
에이전트 A, B, C... 경쟁 입찰 (조건 제시)
    ↓
소비자가 에이전트 선택
    ↓
선택된 에이전트 → 기사 A, B, C...에게 배달 조건 제시
    ↓
기사가 수락
    ↓
배달 완료
    ↓
소비자 평가 (에이전트 + 기사)
```

## 우리가 제공하는 것 (5대 인프라)

안드로이드가 앱 개발자에게 OS를 제공하듯, 우리는 에이전트에게 다섯 가지 인프라를 제공한다.

| # | 인프라 | 설명 |
|---|--------|------|
| 1 | **데이터 API** | 전국 로컬 서비스 데이터 통합 제공 |
| 2 | **결제 처리** | PG 연동 + 크립토 페이먼트 통합 |
| 3 | **웹페이지 (소비자)** | 소비자-에이전트 연결 검색 인터페이스 |
| 4 | **기사 매칭 시스템** | 에이전트-기사 연결 + 배달 조건 입찰 |
| 5 | **기사용 웹페이지** | 기사 대시보드 + 배달 요청 확인/수락 |

에이전트는 추천 알고리즘에만 집중하면 된다. 데이터 수집, 결제 구축, UI 개발을 직접 할 필요가 없다. 진입장벽이 낮아야 에이전트가 많아지고, 에이전트가 많아야 경쟁이 붙는다.

---

## 결제 인프라 (PG + 크립토)

에이전트가 결제까지 각자 구축하면 진입장벽이 높아진다. 결제는 우리가 통합 처리한다.

**결제 수단에 따라 소비자가 지불하는 금액이 달라진다.**

| 결제 수단 | 10,000원 주문 시 | 수수료 |
|-----------|-----------------|--------|
| 카드 결제 | 10,300원 | PG 수수료 ~3% |
| 크립토 결제 (스테이블코인) | 10,000원 | ~0% |

소비자가 결제 수단별 가격 차이를 직접 보고 선택한다. 크립토가 더 싸다는 걸 눈으로 확인하면 자연스럽게 크립토 결제로 이동한다.

## 데이터 수집 (양방향)

### 방향 1: 크롤링
네이버 쇼핑 API + 쿠팡 파트너스 API로 상품 데이터 수집.
- 네이버 플레이스 크롤러 (가게 정보, 메뉴, 가격)
- 카카오맵 크롤러 (가게 정보, 위치)
- 2개 소스 교차검증으로 데이터 신뢰도 확보

### 방향 2: 사업자 직접 등록
음식점, 호텔, 판매자가 직접 데이터를 올리고 판매. 여기에 올리면 에이전트 수백 개에 동시에 노출되므로 배민이나 쿠팡이츠 하나에 입점하는 것보다 이득이다.

생태계가 커질수록 직접 등록 비율이 늘어나면서 크롤링 의존도는 자연스럽게 낮아진다.

## 왜 이기는가

> **같은 음식, 같은 배달, 같은 숙소를 배민·쿠팡이츠에서 10,000원에 사던 것을 여기서는 9,000원에 산다.**

기존 플랫폼이 부과하는 수수료 20%가 사라지면, 절반은 사업자 마진으로, 나머지 절반은 에이전트 경쟁을 통해 소비자 할인으로 돌아간다. 전 영역에서 일관되게 **약 10%의 가격 할인**이 발생한다.

기존 플랫폼은 대응 불가. 수수료를 낮추면 수익 모델이 무너지고, 안 낮추면 소비자가 이탈한다.

## 우선순위 로드맵

### Phase 1: 데이터 확보 (현재)
쇼핑 데이터를 충분히 모아서 쓸만한 서비스를 만든다.

1. 네이버 쇼핑 + 쿠팡 API 크롤링 (상품 데이터 수집)
2. 크롤링 자동 스케줄링 (주기적 수집)
3. 데이터 품질 관리 (중복 제거, 만료 딜 정리)
4. 사업자 직접 등록 기능 (양방향 데이터 수집)

### Phase 2: 에이전트 고도화 + 결제 인프라
에이전트를 진짜 AI로 전환하고 결제를 붙인다.

5. 에이전트별 LLM 연결 (고유 성격 부여)
6. 외부 에이전트 API 공개
7. PG 결제 연동 (통신판매중개업 신고)
8. 크립토 페이먼트 통합 (스테이블코인)

### Phase 3: 영역 확장 (현재)
쇼핑을 넘어 다른 로컬 서비스로 확장한다.

9. ✅ 맛집 데이터 수집 (네이버 플레이스 + 카카오맵 교차검증)
10. ✅ 배달 서비스 (음식 주문 + 에이전트 경쟁 + 기사 매칭)
11. 기사용 웹페이지
12. 숙박 데이터 수집 (야놀자, 에어비앤비)

### Phase 4: 플랫폼 성장
사용자와 에이전트 생태계를 키운다.

13. 모바일 앱 출시
14. 사용자 계정 + 알림 시스템
15. API 과금 시스템
16. 자율주행 로봇 배달 (최종)

## 인프라 에이전트 (Infrastructure Agents)

서비스용 에이전트(소비자 대상)와 별개로, **개발/운영 자동화를 위한 인프라 에이전트**가 동작합니다.

```
┌─────────────────────────────────────────────┐
│  Infrastructure Layer (개발자용)              │
│  코드베이스를 대상으로 동작                    │
└─────────────────────────────────────────────┘
          ↓ 이 위에서 만들어진 코드가
┌─────────────────────────────────────────────┐
│  Service Layer (소비자용)                     │
│  소비자 데이터를 대상으로 동작                 │
└─────────────────────────────────────────────┘
```

### 현재 동작 중

| 에이전트 | 주기 | 역할 | 기록 |
|----------|------|------|------|
| **Maintenance Agent** | 30분 | 코드 논리적 정합성 검사 + 자동 수정 + 빌드 검증 | [CODE_REVIEW_LOG.md](./CODE_REVIEW_LOG.md) |
| **Structure Agent** | 1시간 | 폴더 구조 분석 + 파일 분리/병합 + 네이밍 일관성 검사 | [CODE_REVIEW_LOG.md](./CODE_REVIEW_LOG.md) |
